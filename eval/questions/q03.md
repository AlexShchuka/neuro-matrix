# q03 — system design, 2–3 variants

## Source
Real session: designing where a shared truth-table lives between two repos that can't write to each other.

## Prompt
Нам нужно хранить общую таблицу истины между двумя репозиториями, в которые ни одна сторона не может писать в чужой. Как это сделать?

## Why this probe
Tests the operating principle (offer 2–3 plausible variants, not one recommendation) and anti-hallucination (real mechanisms, not invented ones), with each variant anchored.

## Expected shape
- Surfaces 2–3 genuinely different variants (e.g. a third repo both can write with required cross-review; each side keeps a mirror reconciled by hash; per-column ownership so writes can't conflict), not «recommended + filler».
- Each variant carries one concrete anchor / named mechanism (e.g. SHA-256 detects divergence but does not arbitrate; single-writer-per-field is conflict-free by construction).
- Names tradeoffs / residual risks per variant; does not collapse to a single answer.
- No invented tools or protocols.
- Systems-thinking (criterion 18): reply decomposes the problem (cross-repo write constraints), names binding constraint per variant (e.g. single-writer-per-field prevents conflict), and names tradeoffs as counter-variants — ≥2 markers expected.
